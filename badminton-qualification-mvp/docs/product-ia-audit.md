# Product IA Audit

## 当前结论

当前产品应该被理解为一套“赛事管理与资格核验工作台”，不是单纯的上传记录展示页。

对象层级应保持为：

`赛事 -> 上传批次 -> 核验结果 -> 导出交付物`

其中 `运动员记录` 是核验结果里用于回看和追溯的明细对象，不应被提升为独立主导航对象。

## 应保留的对象层级

- `赛事` 作为一级业务对象，承载名称、日期、主办方、备注
- `上传批次` 作为赛事下的操作对象，承载文件、处理状态、批次留痕
- `核验结果` 作为批次的结果对象，承载风险、复核、匹配明细
- `导出交付物` 作为结果的输出对象，承载 CSV / PDF 等交付结果

## 应删除或收缩的弱价值区块

- 仅用于重复解释产品是什么的说明卡
- 没有真实跳转或真实动作的“快捷操作”入口
- 同时重复表达“风控规则”的多张并列卡片
- 仅展示数字、但不能解释数字来源和下一步动作的统计卡
- 将“上传批次”误写成整个系统主对象的文案

## 文案原则

- 先讲对象，再讲动作，再讲结果
- 标题尽量使用名词短语，避免长句说明书口吻
- 所有入口必须满足三选一：
  - 可点击并真实可用
  - 明确 disabled 并标注即将开放
  - 直接移除
- 默认优先使用“赛事”“上传批次”“核验结果”“导出交付物”四层对象语言
- 少用“历史记录”泛称，必要时只用于“批次留痕”这个页面职责

## 后续页面建议

- 概览页应承担“下一步行动中心”
- 赛事页应承担“赛事创建与编辑”
- 批次留痕页应承担“留痕查询与批次管理”
- 设置页如果暂时没有真实设置能力，应改成“规则与数据说明”或等价职责页

# 集成与验收说明

## 目标

本文件用于对齐 A/B/C/D 四个子代理的联调口径，避免出现前端、后端、数据库和文档各自完成但无法串起来的问题。

## 当前已确认的共享底座

- `ToastProvider` 已挂到根布局，A/B 可以直接复用 `useToast()`。
- `pdfkit` 的服务端打包和中文字体问题已经在当前仓库中处理，不应被其他子代理重复改写。
- `app/dashboard/history/page.tsx` 目前仍是最小历史页，B 需要在此基础上扩展筛选、分页和批量操作。

## 子代理协作边界

- A 只改登录体验相关文件，不碰历史页和数据库。
- B 只改历史记录前端，不直接改 schema。
- C 负责软删除、归档和审计 API，是历史页新交互的后端依赖。
- D 负责把验收口径写清楚，并在各代理完成后做最终联调核对。

## 冲突点预判

- 登录页和历史页都可能需要 toast，但 toast 底座已经共享，避免重复实现。
- 历史页的“删除/归档”如果没有 C 的 API，前端不应硬编码假动作。
- 如果 C 新增 `is_deleted` 或 `archived_at` 字段，历史列表查询必须同步过滤，否则前端会继续看到已删除记录。
- 若 B 把历史页改成客户端分页和筛选，`getHistoryList()` 需要支持服务端过滤参数，否则会先全量拉取再本地筛选，性能和一致性都不理想。

## 验收清单

- 登录页支持一键填充演示账号。
- 登录页支持一键登录、Enter 提交、密码显隐和 loading 状态。
- 登录页布局更偏 SaaS 首页，而不是单纯表单页。
- 历史记录页支持分页、搜索、筛选和批量勾选。
- 删除操作必须走软删除。
- 删除后列表不再显示，但审计信息仍可追溯。
- 风险记录删除时有额外确认或保护。

## 联调步骤

1. 先确认 A 的登录页改造没有破坏 `/api/auth/login` 的登录返回。
2. 再确认 C 的删除/归档 API 已能返回过滤后的历史数据。
3. 然后让 B 接入分页、筛选、批量选择和批量操作。
4. 最后做 D 级验收，逐项核对 UI、接口、审计和边界条件。
